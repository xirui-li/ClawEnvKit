# Contributing: Adding Agent Frameworks

Each agent framework has a different way to discover and use tools. We support three integration approaches.

## Three Approaches

| Approach | When to use | Complexity | Example |
|----------|-------------|------------|---------|
| **A: Native Plugin** | Framework has a plugin/extension system | High | OpenClaw |
| **B: Skill + curl** | Framework has bash/exec capability | Low | NanoClaw, IronClaw, etc. |
| **C: MCP Server** | Framework supports MCP protocol | Medium | Future |

## Approach A: Native Plugin

The agent sees tools like `create_task` natively — same as `sendSlackMessage`. Best UX, no SSRF issues.

**When to use:** The framework has a plugin API with `registerTool()` or equivalent.

**Required files:**

```
extensions/clawharness-{agent}/
├── manifest file (framework-specific format)
├── entry file (TS/JS/Python)
└── package/config file
```

**Core logic (same for all frameworks):**

1. Read `/tmp/eval-tools.json` (generated by entrypoint from OpenAPI spec)
2. For each tool: build parameter schema, register as native tool
3. `tool.execute()` calls `http://localhost:9100/...` directly

**Reference:** See `extensions/clawharness-eval/` (OpenClaw implementation).

## Approach B: Skill + curl

The agent reads a Markdown skill file describing the API, then uses curl via bash/exec.

**When to use:** Most frameworks — anything with shell access.

**Required files:**

1. `docker/Dockerfile.{agent}` — Docker image
2. Config patching in `docker/entrypoint_claw.sh` (add a case for your agent)

No Python adapter needed — all logic is in Docker entrypoints.

**The shared entrypoint handles everything** via environment variables:

| Variable | Purpose | Example |
|----------|---------|---------|
| `AGENT_NAME` | Selects config patching logic | `nanoclaw` |
| `AGENT_CMD` | CLI command to run agent | `nanoclaw agent --local --json` |
| `SKILL_DIR` | Where to write SKILL.md | `~/.nanoclaw/workspace/skills/eval-task` |
| `AGENT_HOME` | Config directory | `~/.nanoclaw` |

**Adding a new Approach B agent:**

### 1. Create Dockerfile

```dockerfile
FROM youragent:latest
USER root
RUN apt-get update && apt-get install -y python3 python3-pip curl jq
RUN pip3 install --break-system-packages fastapi uvicorn pyyaml

COPY clawharness/ /opt/clawharness/clawharness/
COPY mock_services/ /opt/clawharness/mock_services/
COPY docker/entrypoint_claw.sh /opt/clawharness/entrypoint.sh
RUN chmod +x /opt/clawharness/entrypoint.sh

ENV AGENT_NAME=youragent
ENV AGENT_CMD="youragent agent"
ENV SKILL_DIR=/root/.youragent/skills/eval-task
ENV AGENT_HOME=/root/.youragent
# ... standard env vars ...
ENTRYPOINT ["/opt/clawharness/entrypoint.sh"]
```

### 2. Add config patching

In `docker/entrypoint_claw.sh`, add a case to the agent config section:

```python
elif agent == 'youragent':
    config_path = os.path.join(home, 'config.json')
    os.makedirs(os.path.dirname(config_path), exist_ok=True)
    config = {'provider': 'anthropic', 'api_key': api_key, 'model': model}
    with open(config_path, 'w') as f:
        json.dump(config, f, indent=2)
```

## Approach C: MCP Server

Write one MCP server that wraps all mock services. Any MCP-compatible agent can use it.

```
Mock Service (FastAPI) ←HTTP→ MCP Server ←MCP protocol→ Agent
```

**Status:** Planned. See `mcp_servers/clawharness-eval/` (to be created).

## PR Checklist

### New Agent Framework

- [ ] `docker/Dockerfile.{agent}`
- [ ] Config patching in `docker/entrypoint_claw.sh` (if Approach B)
- [ ] OR `extensions/clawharness-{agent}/` (if Approach A)
- [ ] Config patching case in `docker/entrypoint_claw.sh`
- [ ] Update [Supported Agents](../agents/index.md) table
- [ ] Test: `docker run` completes a task with score > 0
